#### 2. 抽象类与接口的区别
共同点:它们都是实现多态和代码复用的核心机制


不同点
- 抽象类可以有==构造方法==，而接口不能有构造方法

- ==实现接口的非抽象类必须要实现该接口的所有方法。抽象类只需要实现全部抽象方法。==

#### 2.1. Jsp和Servlet的区别

- **Servlet**
  - **定义**：Servlet是Java编写的服务器端程序，用于处理HTTP请求并生成动态响应。
  - **角色**：基于==Java类==实现，通过`doGet()`或`doPost()`等方法处理请求，直接操作`HttpServletRequest`和`HttpServletResponse`对象。
  - **生命周期**：由Web容器管理，包括==初始化（`init()`）、处理请求（`service()`）、销毁（`destroy()==`）三个阶段。
- **JSP（JavaServer Pages）**
  - **定义**：JSP是基于HTML的模板技术，允许在HTML中嵌入Java代码（如脚本片段、表达式），简化动态网页开发。
  - **角色**：==本质是Servlet的变体==，JSP文件会被Web容器自动编译为Servlet（如`xxx_jsp.java`），再执行生成响应内容。
  - **生命周期**：JSP转换为Servlet后，生命周期与普通Servlet一致。

|   **特性**   |                    **Servlet**                     |                   **JSP**                    |
| :----------: | :------------------------------------------------: | :------------------------------------------: |
| **设计目的** |  处理业务逻辑、控制请求流程（如MVC的Controller）   |       简化动态页面开发（如MVC的View）        |
| **代码结构** |    纯Java类，通过`response.getWriter()`输出HTML    | HTML为主，嵌入Java代码或JSP标签（如`<% %>`） |
| **开发效率** |     生成==HTML需手动拼接字符串==，开发效率较低     | 直接混合==HTML与Java代码==，适合快速开发页面 |
|  **维护性**  |          逻辑与展示耦合度高，==维护复杂==          |      支持JSTL/EL表达式，分离逻辑与视图       |
| **内置对象** | 需手动获取请求/响应对象（如`request`、`response`） |  自动提供内置对象（如`request`、`session`）  |
| **适用场景** |         控制器、REST API、==复杂业务处理==         |     动态页面渲染（如表格展示、表单提交）     |

#### 2.4  Spring的常用注解(10个)

1. @Controller：定义一个控制器，Spring框架启动时加载，把这个对象交给Spring管理，默认返回视图。

2. @RequestMapping: 指明后端服务器接受到请求时，按何种路径调用方法

3. @RequestParam: 后端参数重命名。例如前端传了一个key为name的变量而后端用createName变量来接受，可以利用该注解实现接收

  ```java
      @RequestMapping("/requestName")
      public String requestName(@RequestParam("name") String createName){
          return "返回的数据为: " +createName;
      }
  ```
4. @RequestBody:接收JSON类型的参数
  ```java
  @RequestMapping("/method1")
  //先创建一个类
      public String method1(@RequestBody Json json){
          return json.toString();//记得重写toString方法
      }
  ```

5. @PathVariable:接收路径参数

```java
       @RequestMapping("/method2/{name}/{id}")
       public String method2(@PathVariable("name") String userName, @PathVariable Integer id){
           return "解析URl数据: " + "userName :"+userName + "   "  +"id: "+ id;
       }
```

6. @Autowired:对类成员变量、方法和构造函数找到与其匹配的Bean实例并进行注入(默认按类型注入)

   - 一个接口有两个实现类

     ```java
     // 1. 定义一个接口
     public interface UserService {
         void sayHello();
     }
     
     // 2. 实现类1：用户服务实现（名称：userServiceImpl）
     @Service // Spring 会默认将该类的 bean 名称设为「类名首字母小写」→ userServiceImpl
     public class UserServiceImpl implements UserService {
         @Override
         public void sayHello() {
             System.out.println("我是用户服务实现类1");
         }
     }
     
     // 3. 实现类2：VIP用户服务实现（名称：vipUserServiceImpl）
     @Service // bean 名称：vipUserServiceImpl
     public class VipUserServiceImpl implements UserService {
         @Override
         public void sayHello() {
             System.out.println("我是VIP用户服务实现类2");
         }
     
     ```
- 注入:

- ```sql
  -- 报错
  @Service
  public class TestService {
      // 错误：Spring 找到 2 个 UserService 类型的 bean（userServiceImpl、vipUserServiceImpl），无法确定注入哪个
      @Autowired
      private UserService userService;
  
      public void test() {
          userService.sayHello();
      }
  }
  
  -- 正确，通过Qualifier指定
  @Service
  public class TestService {
      // 先按类型找 UserService，再按 @Qualifier 指定的名称「userServiceImpl」筛选
      @Autowired
      @Qualifier("userServiceImpl") // 对应实现类1的 bean 名称
      private UserService userService;
  
      public void test() {
          userService.sayHello(); // 输出：我是用户服务实现类1
      }
  }
  ```

#### 2.6  MVC是什么？它的具体实现?

MVC（Model-View-Controller）是一种**软件架构设计模式**，用于将应用程序的逻辑、数据和用户界面分离，提升代码的可维护性和扩展性。以下是其核心概念与典型实现方式：

#####  **一、MVC 的核心组件**

|      组件      |                             职责                             |           示例场景            |
| :------------: | :----------------------------------------------------------: | :---------------------------: |
|   **Model**    |     管理==数据==和==业务逻辑==（如数据库操作、状态管理）     |  用户数据存储、订单计算逻辑   |
|    **View**    |           ==展示数据==（用户界面），不处理业务逻辑           | ==HTML 页面==、移动端 UI 组件 |
| **Controller** | ==接收用户输入==，协调 Model 和 ==View==（如路由请求、调用业务逻辑） |  处理表单提交、API 请求路由   |

##### **二、MVC 工作流程**

1. **用户发起请求**（如点击按钮、访问 URL）

2. **Controller 接收输入**，解析请求参数

3. **Controller 调用 Model** 处理业务逻辑（如查询数据库）

4. **Model 返回数据** 给 Controller

5. **Controller 选择 View** 并传递数据

6. **View 渲染数据**，生成用户可见的界面

  ##### **三、具体实现示例（以 Java Spring MVC 为例）** 

######  **1. Model 层**

处理数据和业务逻辑，通常与数据库交互：

```java
// 数据实体类（POJO）
public class User {
    private Long id;
    private String name;
    // Getters & Setters
}

// 数据访问层（DAO）
@Repository
public class UserRepository {
    public User findUserById(Long id) {
        // 数据库查询逻辑
        return new User(id, "Alice");
    }
}

// 服务层（Service）
@Service
public class UserService {
    @Autowired
    private UserRepository userRepository;
    
    public User getUserDetails(Long userId) {
        return userRepository.findUserById(userId);
    }
}
```

###### **2. Controller 层**

处理 HTTP 请求，协调 Model 和 View：

```java
@Controller
public class UserController {
    @Autowired
    private UserService userService;

    // 处理 GET 请求 /user/123
    @GetMapping("/user/{id}")
    public String getUser(@PathVariable Long id, Model model) {
        User user = userService.getUserDetails(id);
        model.addAttribute("user", user);  // 传递数据到 View
        return "user-profile";  // 返回视图名称（对应 user-profile.html）
    }
}
```

###### **3. View 层**

使用模板引擎（如 Thymeleaf）渲染页面：

```html
<!-- src/main/resources/templates/user-profile.html -->
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<body>
    <h1>用户信息</h1>
    <p>ID: <span th:text="${user.id}"></span></p>
    <p>姓名: <span th:text="${user.name}"></span></p>
</body>
</html>
```

### 2. Mybatis
####  2.1 Mybatis怎么防止SQL注入

##### **一. SQL注入**

SQL注入:一种针对数据库的**安全漏洞攻击手段**，攻击者通过==构造特殊输入==，==欺骗应用程序==执行==非预期的恶意 SQL 语句==，从而窃取、篡改或破坏数据库中的数据。

##### **二.经典攻击示例**

假设登录功能的 SQL 如下：

```sql
SELECT * FROM users 
WHERE username = '[用户输入1]' AND password = '[用户输入2]'
```

若用户输入：

- 用户名：`' OR 1=1 --`
- 密码：任意值（如 `123`）

**最终生成的 SQL**：

```sql
SELECT * FROM users 
WHERE username = '' OR 1=1 -- ' AND password = '123'
```

- `--` 是 SQL 注释符，后续条件被忽略。

- `1=1` 永远成立，攻击者无需密码即可登录任意账户。
#### **二. 预防方法**

##### **1. 优先使用 `#{}` 占位符**

- **原理**：`#{}` 会被 MyBatis 解析为 `PreparedStatement` 的参数占位符（`?`），通过==预编译机制防止 SQL== 注入。

- 示例：

  ```xml
  <!-- 安全示例：使用 #{} -->
  <select id="getUser" resultType="User">
    SELECT * FROM users 
    WHERE username = #{username} AND password = #{password}
  </select>
  ```

  执行逻辑：SQL 会预编译为

  ```
  SELECT * FROM users WHERE username = ? AND password = ?
  ```
  ，参数值通过setString()安全传递。

##### **2. 严格避免直接使用 `${}`**

- 风险：

  ```
  ${}
  ```

  是字符串替换（非预编译），直接拼接 SQL 字符串，存在注入风险。

  ```xml
  <!-- 危险示例：使用 ${} -->
  <select id="getUser" resultType="User">
    SELECT * FROM users 
    WHERE username = '${username}' AND password = '${password}'
  </select>
  ```

  若输入username = ' OR '1'='1

  ，最终 SQL 变为：

  ```sql
  SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '...'
  ```

  结果：绕过密码验证，导致数据泄露。

##### **3. **动态 SQL 的安全写法

##### **1. 使用 `<if>`、`<where>` 标签** 

避免手动拼接 `WHERE` 和 `AND`，防止语法错误和注入：

```xml
<select id="searchUsers" resultType="User">
  SELECT * FROM users
  <where>
    <if test="username != null">
      AND username = #{username}
    </if>
    <if test="email != null">
      AND email = #{email}
    </if>
  </where>
</select>
```

##### **2. 禁止拼接 `ORDER BY` 字段**

- 错误写法

  （存在注入风险）：

  ```xml
  ORDER BY ${sortField} ${sortOrder}
  ```

#### 2.2 Mybatis的一级、二级缓存

#### **一、一级缓存（Local Cache）**

##### **1. 核心特性**

- **作用范围**：`SqlSession` 级别（同一个数据库会话）。
- **默认开启**：无需配置，自动生效。
- **缓存策略**：`SESSION`（默认）或 `STATEMENT`（可通过 `localCacheScope` 配置）。

##### **2. 工作机制**

- **缓存触发**：
  同一 `SqlSession` 中，连续执行**相同的 SQL 和参数**，第二次查询直接返回缓存结果。

  ```java
  SqlSession sqlSession = sqlSessionFactory.openSession();
  UserMapper mapper = sqlSession.getMapper(UserMapper.class);
  
  // 第一次查询数据库
  User user1 = mapper.selectUserById(1L); 
  // 第二次查询一级缓存
  User user2 = mapper.selectUserById(1L); 
  
  System.out.println(user1 == user2); // 输出 true（同一对象）
  ```

- **缓存失效条件**：

  - 执行 `INSERT/UPDATE/DELETE` 操作（任何修改操作）。
  - 手动清空缓存：`sqlSession.clearCache()`。
  - 提交或回滚事务：`sqlSession.commit()` 或 `sqlSession.rollback()`。

##### **3. 配置参数**

```xml
<settings>
  <!-- 可选值：SESSION（默认） / STATEMENT（每次查询后清空缓存） -->
  <setting name="localCacheScope" value="SESSION"/>
</settings>
```

##### **4. 注意事项**

- **对象引用**：一级缓存返回的是**同一个对象**（内存地址相同），修改其中一个会影响缓存数据。

- **分布式环境无效**：仅限单个 `SqlSession`，多实例或分布式系统中无法共享。

#### **二、二级缓存（Second Level Cache）**

  ##### **1. 核心特性**

  - **作用范围**：`Mapper` 级别（跨 `SqlSession` 共享）。
  - **手动开启**：需在 Mapper XML 中显式配置 `<cache/>`。
  - **存储介质**：默认基于内存，可扩展至 Redis、Ehcache 等。

  ##### **2. 配置步骤**

  - **步骤 1：启用全局缓存**（`mybatis-config.xml`）

    ```xml
    <settings>
      <setting name="cacheEnabled" value="true"/> <!-- 默认已开启 -->
    </settings>
    ```

  - **步骤 2：在 Mapper XML 中声明缓存**

    ```xml
    <mapper namespace="com.example.UserMapper">
      <!-- 开启二级缓存 -->
      <cache eviction="LRU" flushInterval="60000" size="1024"/>
      
      <select id="selectUserById" resultType="User" useCache="true">
        SELECT * FROM user WHERE id = #{id}
      </select>
    </mapper>
    ```

  - **步骤 3：实体类实现 `Serializable` 接口**

    ```java
    public class User implements Serializable {
      // 省略字段和方法
    }
    ```

##### **3. 缓存参数详解**

|    **属性**     |                           **说明**                           |
| :-------------: | :----------------------------------------------------------: |
|   `eviction`    | 缓存淘汰策略（默认 `LRU`）：<br>`LRU` - 最近最少使用<br>`FIFO` - 先进先出 |
| `flushInterval` |               缓存刷新间隔（毫秒），默认不刷新               |
|     `size`      |                缓存最大对象数量（默认 1024）                 |
|   `readOnly`    |           是否只读（默认 `false`，可修改缓存对象）           |

##### **4. 工作机制**

- **缓存触发**：
  不同 `SqlSession` 查询相同 Mapper 方法时，若命中缓存则直接返回数据。

  ```java
  // 第一个 SqlSession
  SqlSession sqlSession1 = sqlSessionFactory.openSession();
  User user1 = sqlSession1.getMapper(UserMapper.class).selectUserById(1L);
  sqlSession1.close(); // 必须关闭或提交后，数据才进入二级缓存
  
  // 第二个 SqlSession
  SqlSession sqlSession2 = sqlSessionFactory.openSession();
  User user2 = sqlSession2.getMapper(UserMapper.class).selectUserById(1L); // 从二级缓存获取
  ```

#### **三、一级缓存 vs 二级缓存对比**

|   **特性**   |        **一级缓存**        |            **二级缓存**            |
| :----------: | :------------------------: | :--------------------------------: |
| **作用范围** |     `SqlSession` 内部      |   跨 `SqlSession`（同一 Mapper）   |
| **存储位置** |       内存（JVM 堆）       |     内存或外部存储（如 Redis）     |
| **开启方式** |          默认开启          |       需手动配置 `<cache/>`        |
| **数据共享** |         不可跨会话         |            可跨会话共享            |
| **适用场景** | 高频查询且无并发的单次操作 | 读多写少、数据实时性要求不高的场景 |

####  **3.2. 使用 `free` 命令**

**命令格式**：

```bash
free -h
```

**输出示例**：

```bash
              total        used        free      shared  buff/cache   available
Mem:           7.7G        2.1G        1.2G        200M        4.4G        5.0G
Swap:          2.0G        0B          2.0G
```

**关键列解释**：

- **total**：总内存。
- **used**：已使用的内存（包括缓存和缓冲区）。
- **free**：完全未被使用的内存。
- **buff/cache**：被缓存和缓冲区占用的内存（可快速释放供程序使用）。
- **available**：系统可用内存（= free + buff/cache 可释放部分）。

#### **3.3 使用 `/proc/meminfo` 文件**

**命令格式**：

```bash
cat /proc/meminfo
```

**输出示例**：

```bash
MemTotal:        8058868 kB
MemFree:         1245124 kB
MemAvailable:    5270432 kB
Buffers:          214328 kB
Cached:          4123468 kB
SwapCached:            0 kB
```

**注意事项**：

- **缓存和缓冲区的理解**：Linux 会利用空闲内存做缓存（`buff/cache`），这部分内存可随时释放，因此实际可用内存应参考 `available` 列。
- **Swap 分区的影响**：频繁使用 Swap 可能意味着物理内存不足，需警惕性能问题。

#### 1.IOC（Inversion of Control) 控制反转。是一种设计思想。其思想就是将创建对象的控制权将给Spring框架来管理。控制:创建对象的权力，反转:控制权交给外部环境管理。

### 5. @Autowired与@Resource的区别

1. `@Autowired` 属于 ==Spring 内置的注解==，默认的注入方式为`byType`（根据类型进行匹配），也就是说==会优先根据接口类型去匹配并注入 Bean== （接口的实现类）。

2. `@Resource`属于 JDK 提供的注解，默认注入方式为 ==`byName`==。如果无法通过名称匹配到对应的 Bean 的话，注入方式会变为`byType。

3. `@Resource` 有两个比较重要且日常开发常用的属性：`name`（名称）、`type`（类型）。
4. 如果仅指定 `name` 属性则注入方式为`byName`，如果仅指定`type`属性则注入方式为`byType`，如果同时指定`name` 和`type`属性（不建议这么做）则注入方式为`byType`+`byName`。

### 6.静态内部类和成员内部类的区别

#### 1. 内部类
1. 定义:内部类是指类中的没有被static类修饰的类.==它依赖与外部类的实例==.也就是说必须先创建外部类的实例才能创建该类.其可以访问外部类==所有的成员变量和方法==

2. 内部类示例:

   ```java
   public class OuterClass {
   private int outerData = 1;
   
   public class InnerClass {
   private int innerData = 2;
   
   public void print() {
           System.out.println("Outer data: " + outerData);
           System.out.println("Inner data: " + innerData);
      }
     }
   }
   ```

3. 使用示例

   ```java
   public class InnerClassExample {
   public static void main(String[] args) {
           OuterClass outer = new OuterClass();
           OuterClass.InnerClass inner = outer.new InnerClass();
           inner.print();
   }
   }
   ```

1. 定义:静态内部类是指在类中被static修饰的类.==其实例化不依赖与外部类==.但其==只能访问外部类的静态成员变量和静态方法==.

2. 示例:

3. ```java

   private static int outerData = 1;
   
   public static class InnerClass {
   public void print() {
       System.out.println("Outer data: " + outerData);
       }
     }
   }
   ```

4. 使用示例:

5. ```java
   public class StaticInnerClassExample {
   public static void main(String[] args) {
   OuterClass.InnerClass inner = new OuterClass.InnerClass();

   }
   }
